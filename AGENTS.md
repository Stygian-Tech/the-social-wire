## Learned User Preferences

- Never create git commits or push to remote unless explicitly asked; when asked, stage only files changed for the current task, exclude unrelated edits, use a short conventional commit message, and push the active branch (often `main`).
- Prefer Swift Testing over XCTest for Swift tests (API, worker, packages, iOS) wherever the toolchain supports it.
- Prefer atomic source files: one type or view per file, grouped by folder—not large multi-type files.
- Prefer focused diffs: match existing project style, keep imports at the top, and avoid drive-by refactors or unsolicited docs. When adding Gateway/AppView routes, include matching Bruno requests and OpenAPI drift coverage.
- When a delegated or subagent outcome is already visible in the parent UI, do not restate its full contents unless asked or multi-task synthesis is needed; a brief confirmation is enough (and avoid repeating the same stock confirmation every time).
- When executing an attached implementation plan, do not edit the plan file; use existing to-dos, mark them `in_progress` while working, and avoid creating duplicate to-dos. For features with architectural or product tradeoffs, ask clarifying questions before choosing—do not cross decision points unilaterally.
- Use Title Case for product UI options, menu actions, and button labels unless existing copy clearly follows another pattern.
- Use a confirmation step for bulk read-state actions such as "Mark All As Read" (destructive across many entries). On iOS toolbar bulk-read actions, use `.alert`—not `.confirmationDialog`, which renders as a misaligned popover tail on iPhone.
- For native iOS work, prefer pure SwiftUI with idiomatic system chrome (do not force pixel-perfect mock clones); use UIKit only when SwiftUI cannot meet the requirement or there is a clear performance justification; conform to Swift 6 strict concurrency and Apple Human Interface Guidelines. Account actions such as Log Out belong in ProfileView, opened from the sidebar toolbar avatar—not a sidebar footer.
- Prefer independent distributed backend services (Gateway, AppView, ingestion worker) over a monolithic API, Bluesky/Blacksky-style—not one combined API/AppView process.
- Keep web and iOS as thin consumers: sidebar/feed/folder/avatar/unread logic belongs in Gateway/AppView; do not duplicate discovery, RSS parsing, or projection rules per client. Prefer hard cutover to the server read path over long-lived client-side fallback merge logic. Initial `/read` load uses shared authenticated NDJSON `GET /v1/appview/bootstrap-stream` (Gateway proxies AppView without buffering) for web and iOS—not a client-side sidebar/unread/entries waterfall. User-authored PDS repo writes (folders, subscriptions, prefs, entry read-state records) go direct to the viewer PDS via `PDSClient`/`Agent` with OAuth DPoP; a **401→200 nonce retry** on first write is normal RFC 9449 handshake, not auth failure. AppView mutations (read-mark, mark-all-read, enroll, sidebar resolve/refresh) use Gateway routes. Gateway PDS write-through cannot forward gateway-bound DPoP—supply **`X-ATProto-Upstream-DPoP`** for that path or write direct to PDS; Gateway must also forward that header to AppView on proxied routes that write entry read state to PDS (e.g. mark-all-read). Sidebar unread badges merge AppView baseline with local/PDS read state—not raw server counts alone; mark-all-read clears the full publication-scope badge optimistically (not merely cached-entry deltas). On web, prefetch other publications' entry lists in the background after sidebar load; prefer React Query sidebar projection cache as the bootstrap-stream merge baseline during in-flight loads; bootstrap `unreadCounts` NDJSON is sparse (omits zero-count publications)—apply with `replacePublicationIds` so omitted pubs zero stale badges; use `effectivePublicationUnreadCount` to reconcile server counts with cached feed rows and local read state. On web sidebar load, paint folder/publication section headers immediately with skeleton sub-rows beneath while stream data fills in. Persist sidebar folder/section expand in shared `the-social-wire.sidebar-expanded-keys.v1` per viewer DID (web localStorage, iOS storage).
- On web Base UI `Dialog` modals: reset form state on **open** (not close), keep `DialogContent` a direct child of `Dialog`, use `DialogClose` for Cancel.
- On web, prefer local IndexedDB image caching for publication avatars and entry thumbnails (`imageBlobCache`, `useCachedImageUrl`, `prefetchCachedImages`) so sidebar/feed images are not refetched on every render; dedupe concurrent fetches per URL and prefetch primary `thumbnailUrl` only (fallbacks load on `<img>` error). Display images in whatever format the publisher stored (PDS blob `mimeType` or HTTPS URL)—no client/server transcoding to WebP. Bluesky `cdn.bsky.app` URLs are not directly fetchable from the browser (CORS)—normalize or proxy before caching.

## Learned Workspace Facts

- The Next.js app lives in `apps/web`; common checks are `bun run turbo run build --filter=web...` and targeted `bun test` under `apps/web`. Tests colocate in each owning package (e.g. `apps/web/src/__tests__/`, `services/*/Tests/`, `packages/lexicons/__tests__/`)—not a root `tests/` folder; see `docs/test-plans/`. `bun run dev` runs `next dev --webpack` by default (`dev:turbo` uses Turbopack); keep `turbopack.root` in `next.config.ts` at the monorepo root (`../..`). Only one `next dev` per app directory—kill a stuck `next-server` before deleting `apps/web/.next/dev/lock`. Root metadata in `apps/web/src/app/layout.tsx` sets the default Open Graph/Twitter image to `/og/the-social-wire.png` with `metadataBase` from `NEXT_PUBLIC_SITE_URL`, else `VERCEL_URL` on Vercel, else `https://thesocialwire.app`. The signed-in “My Publications” list is at route `/me/publications` (profile-linked).
- The `/saved` and `/archive` read-later routes use the same three-pane shell as `/read` (list column, detail with `EntryArticleEmbed`; saved/archive layouts keep `main` overflow-hidden; `saved/settings` scrolls inside the page). Archive is a sibling nav destination to Saved (sidebar on web, Lists on iOS)—not only an in-view toggle.
- LATR Link read-later: saved-link **list** uses **`GET /v1/latr/saves`** on latr-gateway (web + iOS; merged display rows built client-side from item records + preview fields). Save/archive/unarchive/delete mutations also use **latr-gateway** (`readLaterProvider` + `latrGatewayClient`, `LatrGatewayClient`; optional web-only `NEXT_PUBLIC_LATR_READ_LATER_PROVIDER=pds-direct` fallback reads/writes PDS directly for debug). Lexicons in `packages/lexicons`, drift-tested against **`latr-packages`** git dep (`LATR_GATEWAY_SAVES_PATH`, record keys, gateway-client upstream DPoP). Official client auth uses server-side credentials on web (`LATR_GATEWAY_CLIENT_CREDENTIAL` or split id/key); iOS uses `LatrGatewayEnvironment`. Saved-link UI shows preview metadata from gateway/PDS enrichment (thumbnail, title, excerpt, site, author). Saved-link list cards show a publication name+favicon chip over the thumbnail (including external saves when resolvable); Bluesky like/reply/repost/quote target the original subject record, not the L@tr saved item. Pocket/getpocket reader saved URLs must embed **`linkedWebUrl`**, not the Pocket reader wrapper (`savedLinkEmbedUrl`).
- Feed entry read/unread uses browser `the-social-wire.read-state.v1` (see `entryReadStateStorage`) and, when signed in, syncs first-read timestamps to `com.thesocialwire.entryReadState` on the user’s PDS (deterministic rkey = base32(SHA-256(entry AT-URI)), same scheme as L@tr item keys). Web and iOS dual-write read marks to AppView and PDS; re-sync on web tab visibility (`useCrossClientReadSync`) and iOS foreground (`syncCrossClientReadState`) by merging PDS markers and refreshing AppView unread counts. OAuth scopes in `auth.ts` + `public/client-metadata.json` must include `repo:com.thesocialwire.entryReadState?...`; widening requires sign-out/in. With the entry list filtered to Unread, defer marking the open entry as read until the user navigates to another entry (avoids the row disappearing while still reading). Sidebar unread badges are scoped per publication, folder, and section, exclude read-later, and show the full count (no 99+ cap). Article embed presentation (feed + read-later): prefer substantial record HTML inline, then oEmbed (`/api/oembed`), then sandboxed iframe for thin RSS-only summaries (`entryArticlePresentation`, `EntryArticleEmbed`); lock mode per article so refetches do not flip; Thin AppView entry detail sets `embedUrl` from `originalUrl` when only a summary exists. Feed and read-later entry lists share vertical detached rounded cards via `articleListCardStyles.ts`; publication chip on saved links only.
- ATProto OAuth uses `atproto` scope: `useAuth().session` is a lightweight `{ did }` shape; anything that needs `fetchHandler` / OAuth on the wire must use `getOAuthSession()` (`OAuthSession`). For the native Swift app, follow ATProto OAuth discovery (`{pds}/.well-known/oauth-protected-resource` → issuer, then `{issuer}/.well-known/oauth-authorization-server`), use mandatory PAR for authorization, exchange tokens at the discovered `token_endpoint` (not ad hoc `{pds}/oauth/token`), serve matching metadata at `apps/web/public/ios-client-metadata.json`, and use native redirect URIs as reversed FQDN with a single slash after the scheme (e.g. `app.example:/oauth/callback`) plus the same scheme in `Info.plist` / `ASWebAuthenticationSession`. Login resolves handles via Bluesky public App View (`com.atproto.identity.resolveHandle`), not the user's PDS; do not restrict `OAUTH_GATEWAY_ALLOWED_AUDIENCES` to a single PDS host.
- **Web env/OAuth:** `getAppEnv()` reads `NEXT_PUBLIC_APP_ENV` or `APP_ENV` (`next.config` forwards server `APP_ENV` to the client bundle). Unset during `next dev` defaults to `local`. Loopback OAuth on localhost applies for `local` and `dev`; hosted dev/preview uses same-origin `/client-metadata.json` with dynamic `redirect_uris` (not prod `public/client-metadata.json`). Production uses discoverable metadata (see `apps/web/src/lib/auth.ts` and README).
- OAuth access tokens here are typically audience-bound to the user’s PDS, not the Bluesky App View—use a plain `Agent('https://bsky.social')` without OAuth-backed `fetch` for public App View reads (e.g. follows, `app.bsky.actor.getProfile`), and use a session-backed `Agent` / `createOAuthAgent` for `com.atproto.repo.*` and other PDS-targeted calls; do not attach the OAuth token to arbitrary `bsky.social` XRPC. When bridging `session.fetchHandler` into `@atproto/api`’s `Agent`, satisfy `typeof fetch` (including static `preconnect`) by delegating `preconnect` from global `fetch` on the wrapper.
- standard.site discovery should cover `site.standard.document` and `site.standard.publication`, and retain `site.standard.entry` for backward compatibility. **Initial load:** authenticated NDJSON **`GET /v1/appview/bootstrap-stream`** (AppView; Gateway-proxied unbuffered)—progressive sidebar slices, unread counts, first-unread auto-selection, first feed page; clients must not invalidate entry queries on stream `done` (page 1 is already seeded); cache-first repeat visits paint persisted cache while the stream refreshes. **Sidebar JSON:** gateway **`GET/POST /v1/publications/sidebar|refresh|resolve`** (implemented in **`services/appview`**, proxied by **`services/gateway`**)—web via **`usePublicationSidebarData`** / **`publicationProjectionClient`** / **`socialWireGatewayClient`**, iOS via **`SocialWireGatewayClient`**. **Add Publication:** gateway **`/v1/publications/resolve`**, then web **`/api/resolve-add-publication`**. RSS branding uses feed icons / favicon fallbacks for sidebar avatars.
- **Distributed Gateway/AppView (Thin AppView hard cutover):** **`services/gateway`** is the public OAuth/DPoP surface and optional PDS write-through (`packages/swift/GatewayCore`, which re-exports **`ATProtoAuthKit`** + **`GatewayTrustKit`** from GitHub)—gateway-bound DPoP proofs cannot be forwarded to PDS; clients may send **`X-ATProto-Upstream-DPoP`** with a PDS-bound proof or write repo records direct to PDS; it proxies reads to **`services/appview`** over HTTP (`APPVIEW_BASE_URL`). **DPoP `htu`:** **`DPoPHtu`** omits the query string (RFC 9449) and prefers **`X-Forwarded-Proto`** over the internal hop scheme behind Fly TLS termination—required for DPoP verification and **DPoP-bound structural** OAuth fallback on routes with query params. **Gateway→AppView trust:** shared **`GATEWAY_APPVIEW_INTERNAL_SECRET`**; **`GatewayInternalTrust`** HMAC signs **path only** (not query) so proxy re-encoding cannot break verification; AppView skips JWT re-verify when trust headers validate; gateway falls back to structural OAuth when issuer JWKS is empty. **`services/appview`** owns sidebar/feed projections, **`/v1/publications/*`**, **`/v1/appview/*`** entry/unread/read-mark routes, and Supabase **`ams`** indexes (`content_items`, read-mark replicas). **`appview_projection_caches`** (Supabase; SQLite on AppView) hold stale-first sidebar/unread/first-page snapshots—invalidate on sidebar refresh, enroll, read-mark, and mark-all-read. **`services/appview-worker`** ingests Jetstream and Skyreader subscriptions; on Linux/Fly, Jetstream firehose uses **WebSocketKit** (`FirehoseLinuxWebSocket`) because URLSession/libcurl cannot open **`wss://`**. **Skyreader RSS feeds are ATProto:** subscriptions live on the user's PDS as **`app.skyreader.feed.subscription`** (Skyreader lexicon)—not a separate non-ATProto model. Worker indexes subscription commits via Jetstream, reads **`feedUrl`** from those records, and parses RSS into **`content_items`** with synthetic author **`did:web:skyreader.rss`** and the normalized feed URL as **`publication_site`**. RSS ingest stable keys prefer normalized article links over opaque `guid` (`RssFeedIdentity` / `rssFeedCore`) so re-polls do not duplicate rows; AppView entry lists dedupe by canonical article URL. Refresh Skyreader timelines via bootstrap ingest, worker RSS poll, or **`feedUrls`** on **`POST /v1/appview/enroll`** (not author enroll). Unread-count and sidebar row lookup must resolve publication ID aliases (URL-encoded `/read/at%3A…` route IDs vs canonical AT-URI)—exact string match can miss counts. **standard.site / Offprint entries:** Thin AppView indexes only PDS records in **`site.standard.document`**, **`com.standard.document`**, **`site.standard.entry`**, **`com.standard.entry`**—not live HTTPS pages alone. Offprint may serve an article at **`https://{host}/a/{rkey}-…`** before (or without) writing the matching **`site.standard.document`** on the author's PDS; **`app.offprint.document.article`** is a strongRef wrapper to that record, not a substitute body. If a subscribed post is visible on the site but missing in-app, check **`com.atproto.repo.getRecord`** / **`listRecords`** on the author PDS first—enroll/backfill cannot index records that were never written. Publication scope matches document **`site`** (AT-URI) or HTTPS **`url`/`siteUrl`** keys from the publication record. Legacy monolith **`services/api`** and **`services/worker`** are removed—do not reintroduce. **PDS canonical** for user-authored records (folders, prefs, subscriptions, read marks); **Supabase only** for rebuildable derived rows (RSS parsed entries, unread counts, enrichment caches). Web/iOS hit the gateway edge URL (`NEXT_PUBLIC_SOCIALWIRE_API_URL` / `SocialWireAPIEnvironment`, still **`api.thesocialwire.app`**): **`NEXT_PUBLIC_USE_THIN_APPVIEW`** (web) / **`SOCIALWIRE_USE_THIN_APPVIEW`** (iOS) gate AppView entry lists; **`POST /v1/appview/enroll`** after sidebar load (client **`recentOnly`** fetches newest page per collection for fast refresh; worker backfill stays full-depth). **`ThinAppViewEntryPagination.aggregate`** (sidebar prefetch **`maxEntries`**) must encode the continuation cursor from the last entry returned after capping, not the last fetched API page cursor. Shared Swift lib **`packages/swift/ThinAppViewCore`**. **Bruno** HTTP collections live under **`services/{gateway,appview,appview-worker}/bruno/`**.
- Bridgy: blob endpoints (`atproto.brid.gy` → `com.atproto.sync.getBlob`) are unreliable for thumbnails—prefer HTTPS normalization and non-Bridgy fallbacks. When normalizing reader/embed URLs, strip `bridge_completed` and similar `bridge_*` query params or publisher iframes can 404.
- **CI/deploy:** Single **`.github/workflows/ci.yml`** (path filters, `needs.changes.result == 'success'`, workflow **`permissions: contents: read`**). Supabase push jobs retry **`actions/checkout`** on transient auth flake (`continue-on-error` + second checkout step). Aggregate job **`CI — required`** gates **`build-web`**, **`test-lexicons`**, **`test-spec`**, Swift **`test-gateway`** / **`test-appview`** / **`test-appview-worker`**, and Supabase jobs. After adding or changing **`packages/*`** workspaces, run root **`bun install`** and commit **`bun.lock`** (CI uses **`--frozen-lockfile`**). Supabase validate/push in CI; use **session pooler** `DATABASE_URL` on GitHub Actions (not direct `db.*.supabase.co`). **No local docker-compose stack** — **`infra/docker`** removed; backend dev is **`swift run`** per service (or Supabase local). Automatic Fly deploy on push to **`dev`**/`**main**`: **gateway**, **appview**, and **appview-worker** when their paths change. Fly apps: **`the-social-wire-*-gateway`**, **`the-social-wire-*-appview`**, **`the-social-wire-*-appview-worker`**; each **`deploy.sh`** calls **`scripts/fly-ensure-app.sh`** to auto-create missing apps. **Fly runtime secrets are per app**—set **`SUPABASE_DATABASE_URL`** (and **`GATEWAY_APPVIEW_INTERNAL_SECRET`** on gateway + appview) on **gateway, appview, and appview-worker**; monolith secrets do not carry over. Prefer **session pooler** for Fly **`SUPABASE_DATABASE_URL`** too—direct **`db.*.supabase.co`** often fails from Fly (connection refused / IPv6); AppView **`28P01`** / user **`postgres`** usually means malformed URL (URL-encode password with **`%20`** for spaces, not **`+`**; username **`postgres.[ref]`** for pooler); compare **`fly secrets list`** digests when values look identical across apps. Deploy from repo root — Docker **build context = monorepo root**. **GitHub:** **`FLY_API_TOKEN`** required; do **not** rely on legacy **`FLY_APP_DEV`** / **`FLY_WORKER_APP_*`** (removed from deploy fallbacks). **Custom domains** **`api.testing.thesocialwire.app`** / **`api.thesocialwire.app`** must attach to **`*-gateway`** Fly apps via **`flyctl certs add`** (not legacy **`the-social-wire-dev`**); stale **AAAA** records can cause **`ERR_CONNECTION_RESET`** when browsers prefer IPv6. **Linux Swift (CI/Fly):** shared **`packages/`/`services/`** use **`import Crypto`** (swift-crypto), not **`CryptoKit`**; **`RssFeedParser`** imports **`FoundationXML`** with **`#if !canImport(Darwin)`** — **`#if canImport(FoundationXML)`** is false in SPM on Linux. **Fly runtime deps:** gateway/appview **`libssl3 libcurl4 libsqlite3-0 libicu70 libxml2`**; appview-worker **`libssl3 libcurl4 libsqlite3-0 libxml2`**; Docker build stages need **`libxml2-dev`** (ThinAppViewCore **`FoundationXML`** dynamically links **`libxml2.so.2`**). Wiki sync **main** only (`publish-wiki.yml`). Manual redeploy: **`deploy.yml`** `workflow_dispatch`.
- **Native iOS app:** `SocialWireAPIEnvironment` targets **`https://api.testing.thesocialwire.app`** when **`DEBUG`** or **`SOCIALWIRE_TESTING_API`** is set, else production **`https://api.thesocialwire.app`**; **`client_id`** is **`iosClientMetadataURL`** on that host. **TestFlight** is usually **`Release`** without **`DEBUG`**—set **`SOCIALWIRE_TESTING_API`** for beta when testing API/metadata is required. Web and iOS read preferences via authenticated **`GET /v1/sync/preferences`** (not direct browser/PDS `getRecord`); also use **`/v1/pds/cache/record`** where applicable. Initial reader load uses the same **`/v1/appview/bootstrap-stream`** NDJSON contract as web. Legacy **`/discovery`** / **`/entries`** need **`ENABLE_LEGACY_CONTENT_API`** and are not the default iOS path. On-device reader cache uses **SwiftData** (`ReaderCacheCoordinator`, `ReaderSwiftDataModels`), not GRDB. No root **`Package.swift`**—build iOS via Xcode (`apps/apple`); run **`swift build`**/**`swift test`** per **`services/*`** or **`packages/swift/*`** directory. **iPhone compact:** horizontal swipe **`TabView`** **`.page`** with four panes (Lists → Publications → Articles → Reader); **Lists** holds Read Later / Archive / Subscribed / Following; reopen to **last selected list** in **`UserDefaults`**; animate pane changes on selection like swipe; selecting a publication lands on **Articles** (not **Reader**); compact publication rows need an explicit tap handler—`List`/`onChange` will not re-fire when bootstrap or cache pre-selected the row; resolve publications via **`publication(forId:)`** across all sidebar arrays, not **`allPublicationRows`** only; swiping Articles→Publications clears **`selectedSidebar`** (keep **`selectedPublication`**) so re-tapping the same publication re-navigates to Articles; use clear **`List`** backgrounds and minimal chrome on the bottom **All/Unread** picker. **Reader chrome** is hosted once on the compact pager (toolbar on inner **`NavigationStack`**, not duplicated per pane): **Mark All As Read** as **`checkmark.circle`** in **topBarTrailing**, scoped to the active pane (all lists, one list source, one publication’s articles, or the focused reader entry including unread-deferred feed rows); **All/Unread** segmented **`Picker`** in a fixed-height bottom **`safeAreaInset`** across all four panes (prevents **`ContentUnavailableView`** jump). **Subscribed** uses collapsible **`List`** sections for folders/publications; **Add Publication** only on **Subscribed** (not **Following**). Empty states: centered **`ContentUnavailableView`** with **"Select a Publication"** / **"Select an Article"**. **Regular width:** **`NavigationSplitView`** (sidebar | list | reader). Profile avatar in sidebar **toolbar** opens **ProfileView** (Log Out there). OAuth/prod-vs-testing skew (`invalid_client_metadata`, `invalid_redirect_uri`) usually means **`client_id`** / **`redirect_uris`** mismatch vs the active API host.
