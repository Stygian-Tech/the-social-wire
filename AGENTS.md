## Learned User Preferences

- When asked to commit and push, stage only files changed for the current task, exclude unrelated edits, use a short conventional commit message, and push the active branch (often `main`).
- Prefer focused diffs: match existing project style, keep imports at the top, and avoid drive-by refactors or unsolicited docs.
- When a delegated or subagent outcome is already visible in the parent UI, do not restate its full contents unless asked or multi-task synthesis is needed; a brief confirmation is enough (and avoid repeating the same stock confirmation every time).
- When executing an attached implementation plan, do not edit the plan file; use existing to-dos, mark them `in_progress` while working, and avoid creating duplicate to-dos.
- Use Title Case for product UI options, menu actions, and button labels unless existing copy clearly follows another pattern.
- Use a confirmation step for bulk read-state actions such as "Mark All As Read" (destructive across many entries).
- For native iOS work, prefer pure SwiftUI with idiomatic system chrome (do not force pixel-perfect mock clones); use UIKit only when SwiftUI cannot meet the requirement or there is a clear performance justification; conform to Swift 6 strict concurrency and Apple Human Interface Guidelines. Account actions such as Log Out belong in ProfileView, opened from the sidebar toolbar avatar—not a sidebar footer.

## Learned Workspace Facts

- The Next.js app lives in `apps/web`; common checks are `bun run turbo run build --filter=web...` and targeted `bun test` under `apps/web`. Root metadata in `apps/web/src/app/layout.tsx` sets the default Open Graph/Twitter image to `/og/the-social-wire.png` with `metadataBase` from `NEXT_PUBLIC_SITE_URL`, else `VERCEL_URL` on Vercel, else `https://thesocialwire.app`. The signed-in “My Publications” list is at route `/me/publications` (profile-linked).
- The `/saved` read-later route uses the same three-pane shell as `/read` (list column, detail with `EntryArticleEmbed`; `saved/layout` keeps `main` overflow-hidden and `saved/settings` scrolls inside the page).
- LATR Link HTTPS read-later merges and persists via `com.latr.saved.*` records on the user’s PDS (lexicons in `packages/lexicons`).
- Feed entry read/unread uses browser `the-social-wire.read-state.v1` (see `entryReadStateStorage`) and, when signed in, syncs first-read timestamps to `com.thesocialwire.entryReadState` on the user’s PDS (deterministic rkey = base32(SHA-256(entry AT-URI)), same scheme as L@tr item keys). OAuth scopes in `auth.ts` + `public/client-metadata.json` must include `repo:com.thesocialwire.entryReadState?...`; widening requires sign-out/in. With the entry list filtered to Unread, defer marking the open entry as read until the user navigates to another entry (avoids the row disappearing while still reading). Sidebar unread badges are scoped per publication, folder, and section, exclude read-later, and show the full count (no 99+ cap).
- ATProto OAuth uses `atproto` scope: `useAuth().session` is a lightweight `{ did }` shape; anything that needs `fetchHandler` / OAuth on the wire must use `getOAuthSession()` (`OAuthSession`). For the native Swift app, follow ATProto OAuth discovery (`{pds}/.well-known/oauth-protected-resource` → issuer, then `{issuer}/.well-known/oauth-authorization-server`), use mandatory PAR for authorization, exchange tokens at the discovered `token_endpoint` (not ad hoc `{pds}/oauth/token`), serve matching metadata at `apps/web/public/ios-client-metadata.json`, and use native redirect URIs as reversed FQDN with a single slash after the scheme (e.g. `app.example:/oauth/callback`) plus the same scheme in `Info.plist` / `ASWebAuthenticationSession`.
- **Local `next dev`:** parameterized loopback OAuth builds `http://localhost?redirect_uri=…` (with `127.0.0.1`/`[::1]` and `/callback`) when `NEXT_PUBLIC_APP_ENV` is `local` or unset in development; production uses discoverable client metadata (see `apps/web/src/lib/auth.ts` and README).
- OAuth access tokens here are typically audience-bound to the user’s PDS, not the Bluesky App View—use a plain `Agent('https://bsky.social')` without OAuth-backed `fetch` for public App View reads (e.g. follows, `app.bsky.actor.getProfile`), and use a session-backed `Agent` / `createOAuthAgent` for `com.atproto.repo.*` and other PDS-targeted calls; do not attach the OAuth token to arbitrary `bsky.social` XRPC. When bridging `session.fetchHandler` into `@atproto/api`’s `Agent`, satisfy `typeof fetch` (including static `preconnect`) by delegating `preconnect` from global `fetch` on the wrapper.
- standard.site discovery should cover `site.standard.document` and `site.standard.publication`, and retain `site.standard.entry` for backward compatibility; `services/api`’s Swift `DiscoveryChain.swift` reflects the intended multi-collection behavior. In the web app, sidebar subscribed sources merge `site.standard.graph.subscription` with Skyreader `app.skyreader.feed.subscription` records (`usePublicationSidebarData`, `pdsClient`). The Add Publication flow (`AddPublicationDialog` + `/api/resolve-add-publication`) prefers subscribing to a resolved standard.site publication when possible, otherwise falls back to RSS/Atom and writes a Skyreader feed subscription; RSS branding resolves feed icons from `channel.image.url` / `itunes:image`, stores them as `customIconUrl`, and uses site/feed-origin favicon fallbacks for sidebar avatars.
- Hosted `services/api` (`api.thesocialwire.app` / testing) is for **first-party clients only**, not a public integration API for arbitrary apps; enforce via `OAUTH_GATEWAY_ALLOWED_CLIENT_IDS`, `OAUTH_GATEWAY_ALLOWED_AUDIENCES`, and `OAUTH_GATEWAY_REQUIRE_KNOWN_CLIENT` (see `services/api/README.md`). Others may self-host their own gateway with their own allowlists.
- **Thin AppView** (`ENABLE_THIN_APPVIEW` on gateway + Fly **`worker`** process): Level-1 entry index (`content_items`) and derived **`read_marks`** in Supabase **`ams`**; authenticated **`/v1/appview/*`** routes for entry timelines, read-mark write-through, enrollment backfill, and privacy purge. Web: **`NEXT_PUBLIC_USE_THIN_APPVIEW`** + **`thinAppViewClient.ts`** (PDS remains canonical for writes and entry detail). iOS: compile flag **`SOCIALWIRE_USE_THIN_APPVIEW`** on **`SocialWireGatewayClient`**. Bluesky **`public.api.bsky.app`** usage is unchanged. After publication discovery, clients call **`POST /v1/appview/enroll`** with followed author DIDs.
- Bridgy blob endpoints (`atproto.brid.gy` → `com.atproto.sync.getBlob`) are unreliable for thumbnails—responses may `400`, and `http://` variants trigger mixed-content upgrades—prefer HTTPS normalization and non-Bridgy fallbacks rather than a single Bridgy blob URL as the only `<img>` source. When normalizing external reader/embed URLs, strip Bridgy-related query noise such as `bridge_completed` (and similar `bridge_*` params); leaving them on can yield publisher-site `404`s in iframes or outbound navigation.
- In `apps/web`, `bun run dev` runs `next dev --webpack` by default (`bun run dev:turbo` uses Turbopack). Keep `turbopack.root` in `next.config.ts` at the monorepo root (`../..`) so `next build` resolves the workspace and Turbopack does not infer a lockfile under `$HOME`; setting the root to `apps/web` alone breaks production builds. Next allows only one `next dev` per app directory—a stuck `next-server` or stale `apps/web/.next/dev/lock` can block startup; stop the old process (e.g. `kill -9` the listening PID) before deleting the lock if needed.
- **Native iOS app:** `SocialWireAPIEnvironment` targets **`https://api.testing.thesocialwire.app`** when **`DEBUG`** or **`SOCIALWIRE_TESTING_API`** is set, else production **`https://api.thesocialwire.app`**; **`client_id`** is **`iosClientMetadataURL`** on that host. **TestFlight** is usually **`Release`** without **`DEBUG`**—set **`SOCIALWIRE_TESTING_API`** for beta when testing API/metadata is required. Prefer authenticated **`/v1/sync/preferences`** and **`/v1/pds/cache/record`**; legacy **`/discovery`** / **`/entries`** need **`ENABLE_LEGACY_CONTENT_API`** and are not the default iOS path. On-device reader cache uses **SwiftData** (`ReaderCacheCoordinator`, `ReaderSwiftDataModels`), not GRDB. **iPhone compact:** horizontal swipe **`TabView`** **`.page`** with four panes (Lists → Publications → Articles → Reader); **Lists** holds Read Later / Subscribed / Following; reopen to **last selected list** in **`UserDefaults`**; animate pane changes on selection like swipe. **Reader chrome** is hosted once on the compact pager (toolbar on inner **`NavigationStack`**, not duplicated per pane): **Mark All As Read** as **`checkmark.circle`** in **topBarTrailing**, scoped to the active pane (all lists, one list source, one publication’s articles, or the focused reader entry including unread-deferred feed rows); **All/Unread** glass segmented **`Picker`** in a fixed-height **`safeAreaInset`** bottom bar on **Publications** and **Articles** (prevents **`ContentUnavailableView`** jump). **Subscribed** uses collapsible **`List`** sections for folders/publications; **Add Publication** only on **Subscribed** (not **Following**). Empty states: centered **`ContentUnavailableView`** with **"Select a Publication"** / **"Select an Article"**. **Regular width:** **`NavigationSplitView`** (sidebar | list | reader). Profile avatar in sidebar **toolbar** opens **ProfileView** (Log Out there). OAuth/prod-vs-testing skew (`invalid_client_metadata`, `invalid_redirect_uri`) usually means **`client_id`** / **`redirect_uris`** mismatch vs the active API host.
